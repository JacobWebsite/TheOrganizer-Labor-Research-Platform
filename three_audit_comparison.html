How to Read This Document

Three different AI systems each did a "blind" audit of the platform ‚Äî meaning none of them saw each other's work. This document compares what they found.

Think of it like getting three independent inspectors to look at a house: where they all agree, you can be very confident those issues are real. Where only one noticed something, it might be a unique insight ‚Äî or it might be a stretch.

"Orphaned rows" = Data that points to records that no longer exist (like an address book entry for a demolished building)
"CORS" = A security setting controlling which websites can talk to your system. Wide open = anyone can access it.
"Auth" = Authentication ‚Äî requiring people to log in before using the system
"Match rate" = The percentage of records from one database successfully linked to records in another

üî¥ Security: The Platform Is Not Safe for Shared Use

üî¥ Password Exposed in Code

üî¥ The 50% Orphaned Data Problem

What's wrong: When the system links unions to employers, about half of those links (60,373 out of ~120,000) point to employer records that no longer exist. This happened because the system went through a cleanup process but the links weren't updated to point to the new clean records.

Why it matters: Imagine a phone directory with 120,000 entries where 60,000 numbers have been changed but nobody updated the directory. If someone looks up a union and asks "which employers do they bargain with?", the system silently skips half the answers. No error message ‚Äî just an incomplete list. Every search, every score, every analysis is working with only half the real information.

Part 3: Unique Catches

Issues only one auditor found ‚Äî potential unique insights or blind spots.

Claude Only Claude Found

  1. LIMIT 500 pre-filter bug in scoring ‚Äî The system grabs 500 employers sorted by violations, then scores those 500. But an employer with mediocre violations but great scores on other factors never gets evaluated. The system finds "top 500 by one measure" instead of "top 500 overall."
  2. Rate limiter memory leak ‚Äî The system tracks internet addresses that made requests but never cleans up old entries. Over days/weeks, this list grows forever and slowly eats memory.
  3. Building trades over-count by 10-32x ‚Äî F7 reports count "covered workers" (everyone on a project), while LM2 counts dues-paying members. Building trade numbers are wildly inflated and the platform doesn't explain this.
  4. GLEIF data ROI problem ‚Äî One dataset takes 10.6 GB (over HALF the database) but only produced 605 matches. Enormous storage for very little value.
  5. Scorecard ceiling for unmatched employers ‚Äî 86% of OSHA establishments can't link to union data, so they automatically lose points. Most scores get squeezed into 0-50 out of 100, making tiers less meaningful.
  6. Two separate, unlabeled scoring systems ‚Äî An OSHA scorecard (0-100) and a Mergent sector scorecard (0-62). A score of "45" could come from either with no way to tell.

Gemini Only Gemini Found

  1. FMCS Contract Expiration Data as highest priority new source ‚Äî When union contracts are about to expire, that's a critical organizing moment. Gemini identified this as the single most valuable missing dataset.
  2. OpenSecrets.org API as alternative to raw FEC data ‚Äî Use pre-cleaned political contribution data instead of processing the messy raw files.
  3. News and media monitoring elevated in priority ‚Äî Real-time alerts about strikes, layoffs, etc. moved from wish list to medium-term priority.
  4. MLOps practices for predictive modeling ‚Äî Experiment tracking, model registries, model monitoring ‚Äî infrastructure for running machine learning seriously.
  5. Splink as central matching strategy ‚Äî Make Splink the primary approach for ALL data matching, replacing the current tier system.
  6. Database migration tooling (Alembic/Flyway) ‚Äî A system for tracking database structure changes over time.

Codex Only Codex Found

  1. 6 foreign keys without supporting indexes ‚Äî Specific database columns with relationship constraints but no fast-lookup indexes. Will cause slowdowns as data grows.
  2. Frontend hardcoded to localhost ‚Äî The web interface has localhost:8001 written directly in the code. Must be configurable for any deployment.
  3. Score model versioning ‚Äî Store a version number with each scoring run so you can track when scores change due to formula updates vs. data changes.
  4. Orphan solution: split into current vs. historical ‚Äî Instead of re-linking orphaned data, split the table into "current" and "historical" so queries always know which era they're in.
  5. "Evidence packet" export concept ‚Äî Generate a printable bundle of all evidence about a target employer (safety, wage theft, elections, comparables) for campaign use.

Part 6: Key Takeaways

The Three Auditors Paint a Consistent Picture

  1. The data foundation is impressive and real. 14.5 million deduplicated union members, validated within 1.4% of federal benchmarks. This is not a toy project.
  2. The platform is a working prototype, not a production system. It works for a single researcher locally. It's NOT ready for multiple users on a shared server.
  3. The single highest-impact fix is the orphaned data problem. Fixing 60,373 broken links improves every output. (Gemini missed this ‚Äî notable.)
  4. Security must be addressed before sharing. All three agree: enable login, restrict access, change the password.
  5. Matching and scoring need automated tests. The intellectual core of the platform has zero safety nets.
  6. The frontend works but is fragile. Breaking it up is important but not urgent.
